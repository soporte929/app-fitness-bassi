# Plan 17.1: Migrate to next-themes

<objective>
Replace the custom ThemeProvider and FOUC-prevention inline script with the `next-themes` library. The toggle component already exists in both client header and trainer sidebar — this plan rewires it to use next-themes.

Purpose: Eliminates hand-rolled theme logic, prevents hydration mismatches out-of-the-box, and sets the foundation for Plan 17.2 (hardcoded color cleanup).
Output: Working dark/light toggle backed by next-themes with localStorage persistence and zero FOUC.
</objective>

<context>
Load for context:
- .planning/ROADMAP.md (Phase 17 requirements)
- app/layout.tsx (current inline script + ThemeProvider wrapper)
- components/providers/theme-provider.tsx (custom implementation to replace)
- components/ui/theme-toggle.tsx (toggle component to rewire)
- app/globals.css (CSS var definitions — `:root` dark, `[data-theme="light"]` light)
</context>

<task type="auto">
  <name>Replace ThemeProvider and remove FOUC script</name>
  <files>
    components/providers/theme-provider.tsx
    app/layout.tsx
  </files>
  <action>
    1. **Rewrite `components/providers/theme-provider.tsx`:**
       - Import `ThemeProvider as NextThemesProvider` from `next-themes`.
       - Export a `ThemeProvider` component that wraps children in `<NextThemesProvider attribute="data-theme" defaultTheme="dark" enableSystem={false} disableTransitionOnChange={false}>`.
       - IMPORTANT: Use `attribute="data-theme"` because the CSS in `globals.css` uses `[data-theme="light"]`. Also add `attribute` as array `{["class", "data-theme"]}` so that BOTH `class="dark"` and `data-theme="dark"` are set — the Tailwind `@custom-variant dark (&:is(.dark *))` in globals.css depends on the `dark` class.
       - Re-export `useTheme` from `next-themes` so existing imports from this file still work.
       - Keep the file as `'use client'`.

    2. **Update `app/layout.tsx`:**
       - Remove the `<head>` block containing the inline `<script dangerouslySetInnerHTML>` — next-themes injects its own script automatically.
       - Remove `data-theme="dark"` from the `<html>` tag — next-themes manages this.
       - Keep `suppressHydrationWarning` on `<html>` (next-themes requires it).
       - Keep the `dark` class in the className for SSR initial render — next-themes will manage it from there.
       - Actually, remove the hardcoded `dark` from className too — next-themes adds the class automatically based on defaultTheme. Keeping it could cause a mismatch. The `suppressHydrationWarning` handles the SSR → client transition.

    AVOID: Do NOT set `enableSystem={true}` — the spec says user-persisted preference only, no OS detection.
    AVOID: Do NOT set `storageKey` — the default `"theme"` matches what was used before, ensuring existing users keep their preference.
  </action>
  <verify>
    - Open the app in browser, check console for hydration warnings (must be zero).
    - Inspect `<html>` element: should have `class="dark"` AND `data-theme="dark"` initially.
    - Toggle theme: `<html>` switches to no `dark` class + `data-theme="light"`.
    - Refresh: persisted choice survives.
  </verify>
  <done>ThemeProvider uses next-themes, the FOUC script is gone, and the html element gets both `class` and `data-theme` attributes managed by next-themes.</done>
</task>

<task type="auto">
  <name>Rewire ThemeToggle to next-themes</name>
  <files>components/ui/theme-toggle.tsx</files>
  <action>
    1. Change the import from `@/components/providers/theme-provider` to `next-themes` (or keep importing from `@/components/providers/theme-provider` since it re-exports `useTheme`).
    2. Use `const { resolvedTheme, setTheme } = useTheme()` from next-themes.
    3. Replace `toggleTheme` call with `setTheme(resolvedTheme === 'dark' ? 'light' : 'dark')`.
    4. Use `resolvedTheme` (not `theme`) for the icon display — `resolvedTheme` is always 'dark' or 'light' even during hydration.
    5. During SSR/mount, `resolvedTheme` may be undefined — render a neutral placeholder (empty `<div className="w-4 h-4" />`) when undefined to avoid layout shift. Use `const [mounted, setMounted] = useState(false)` + `useEffect → setMounted(true)` pattern.

    AVOID: Do NOT use `theme` from next-themes for rendering icons — it can be 'system' which doesn't map to an icon.
  </action>
  <verify>
    - Click the toggle: icon switches between Sun and Moon.
    - Refresh: icon matches the persisted theme.
    - No React hydration mismatch in console.
  </verify>
  <done>ThemeToggle uses next-themes, shows correct icon, and has no hydration issues.</done>
</task>
