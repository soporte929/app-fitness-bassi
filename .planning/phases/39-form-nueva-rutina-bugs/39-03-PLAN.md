# Plan 39.3: Fix Select "Días por semana" pierde valor al volver a Step 1

<objective>
Corregir el bug donde el select "Días por semana" en Step 1 del RoutineBuilder pierde su valor visible (queda vacío) al navegar a Step 2 y regresar a Step 1, aunque el valor interno (`state.daysPerWeek`) sigue correcto.

Purpose: El usuario pierde confianza en el formulario cuando un campo ya rellenado aparece vacío.
Output: `components/ui/select.tsx` corregido para que los labels persistan correctamente entre re-renders.
</objective>

</context>

<task type="auto">
  <name>Fix label persistence in custom Select component</name>
  <files>components/ui/select.tsx</files>
  <action>
    **Root cause analysis:**
    
    El componente `Select` (custom, no Radix) almacena labels en un `useRef(new Map())`. Los `SelectItem` registran sus labels llamando `registerLabel(value, children)` directamente durante el render (línea 146-148 del select.tsx). `SelectValue` lee el label con `getLabel(value)`.

    Tras el fix de 39-01, los pasos del wizard ya NO se desmontan/remontan — se ocultan con CSS `display: none`. Esto significa:

    1. En el render inicial, TODOS los steps se montan a la vez incluido el Select y sus SelectItems.
    2. `registerLabel` se llama durante el render síncrono y llena el Map.
    3. `SelectValue` lee del Map y muestra el label correctamente.
    4. Al navegar a Step 2 (CSS hidden) y volver a Step 1 (CSS block), React NO re-renderiza los children porque no hubo cambio de props/state.
    5. **Problema**: Si React triggerea un re-render del Select (por cualquier motivo — parent re-render, context change), `SelectValue` intenta leer del Map PERO `SelectItem` podría no haber sido re-renderizado en ese ciclo si el `SelectContent` usa `{open && ...}` para el backdrop — aunque el dropdown mismo usa CSS hidden, la interacción de re-renders puede causar que el Map no tenga la entrada en el momento que SelectValue la necesita.

    Alternativa más probable: cuando el step contenedor tiene `display: none` y luego vuelve a `display: block`, el browser puede forzar un repaint que, combinado con React strict mode o batching, causa que `SelectValue` se re-renderice antes de que los `SelectItem` hayan re-ejecutado `registerLabel`.

    **Fix approach — hacer el label registry más robusto:**
    
    Opción elegante: Mover el registro de labels a un `useEffect` en `SelectItem` en vez de hacerlo durante el render. Usar un state Map en vez de un ref Map para que los cambios en labels triggeen re-renders de `SelectValue`.

    Implementación concreta:
    
    1. En `Select`, cambiar `const labels = useRef(new Map())` por `const [labels, setLabels] = useState(new Map<string, string>())`.
    2. Actualizar `registerLabel` para usar `setLabels` con la functional updater:
       ```ts
       const registerLabel = useCallback((v: string, label: string) => {
         setLabels(prev => {
           if (prev.get(v) === label) return prev  // skip if unchanged
           const next = new Map(prev)
           next.set(v, label)
           return next
         })
       }, [])
       ```
    3. Actualizar `getLabel` para leer del state en vez del ref:
       ```ts
       const getLabel = useCallback((v: string) => labels.get(v), [labels])
       ```
    4. En `SelectItem`, mover el `registerLabel` call a un `useEffect` para evitar llamar setState durante render:
       ```ts
       useEffect(() => {
         if (typeof children === 'string') {
           registerLabel(value, children)
         }
       }, [value, children, registerLabel])
       ```
       Eliminar el call síncrono `if (typeof children === 'string') { registerLabel(value, children) }` del cuerpo del render.

    AVOID: No cambiar la API pública de Select/SelectItem/SelectValue/SelectTrigger/SelectContent. Mantener compatibilidad con todos los demás usos de Select en el proyecto.
    AVOID: No usar `useLayoutEffect` — no es necesario y genera warnings en SSR.
  </action>
  <verify>

    4. Cambiar a "5 días" → Navegar a Step 2 → Volver a Step 1 → Sigue mostrando "5 días".
    5. Verificar que el select funciona correctamente en otros contextos si existen.
  </verify>
