# Plan 33.1: Verificar migración y forzar reload de schema cache en producción

## Objective
Resolver el error `Could not find the 'diet_type' column of 'nutrition_plans' in schema cache` que rompe `/nutrition-plans` y el modal "Crear plantilla" en producción. El problema es que las columnas `diet_type`, `meals_count` e `is_template` fueron añadidas en la migración Phase 10.1 pero el schema cache de PostgREST en producción no se ha actualizado.

## Context
- `.planning/ROADMAP.md` — Phase 33 description
- `supabase/migrations/20260309223820_nutrition_plan_metadata.sql` — Migration SQL (ALTER TABLE con 3 columnas)
- `lib/supabase/types.ts` — TypeScript types (ya incluyen diet_type, meals_count, is_template)
- `app/(trainer)/nutrition-plans/actions.ts` — Server actions que referencian diet_type, is_template
- `app/(trainer)/nutrition-plans/page.tsx` — Page que query `.eq('is_template', true)`
- `.env.local` — Supabase URL: `https://hmsebbkfcsaotdgoltnf.supabase.co`

<task type="auto">
  <name>Verificar que la migración existe en producción</name>
  <files>supabase/migrations/20260309223820_nutrition_plan_metadata.sql</files>
  <action>
    1. Abrir el dashboard de Supabase para el proyecto `hmsebbkfcsaotdgoltnf`
       URL: https://supabase.com/dashboard/project/hmsebbkfcsaotdgoltnf/editor
    2. En el SQL Editor, ejecutar:
       ```sql
       SELECT column_name, data_type, is_nullable
       FROM information_schema.columns
       WHERE table_name = 'nutrition_plans'
       AND column_name IN ('diet_type', 'meals_count', 'is_template')
       ORDER BY column_name;
       ```
    3. **Si devuelve 3 filas:** La migración ya está aplicada → ir a Task 2 (reload cache).
    4. **Si devuelve 0 filas:** La migración NO se aplicó → ejecutar manualmente el SQL:
       ```sql
       ALTER TABLE public.nutrition_plans
       ADD COLUMN IF NOT EXISTS diet_type text CHECK (diet_type IN ('A', 'B', 'C')),
       ADD COLUMN IF NOT EXISTS meals_count integer DEFAULT 3,
       ADD COLUMN IF NOT EXISTS is_template boolean DEFAULT false;
       ```
       Verificar que el ALTER TABLE retorna sin error.
    - Usar `IF NOT EXISTS` para que sea idempotente y evitar errores si alguna columna ya existe.
    - NO modificar la constraint CHECK existente ni los defaults si las columnas ya están.
  </action>
  <verify>
    Ejecutar la query de information_schema y confirmar que devuelve exactamente 3 filas:
    - diet_type (text)
    - meals_count (integer)
    - is_template (boolean)
  </verify>
  <done>Las 3 columnas (diet_type, meals_count, is_template) existen en la tabla nutrition_plans de producción</done>
</task>

<task type="auto">
  <name>Forzar reload del schema cache de PostgREST</name>
  <files>N/A (operación en Supabase dashboard)</files>
  <action>
    1. En el SQL Editor de Supabase, ejecutar:
       ```sql
       NOTIFY pgrst, 'reload schema';
       ```
       Esto fuerza a PostgREST a invalidar su schema cache y re-leer el esquema de la DB.
    2. Alternativamente, si `NOTIFY` no surte efecto:
       - Ir a Settings → API en el dashboard de Supabase
       - Clic en "Reload schema" o reiniciar el API service
    3. Esperar ~10 segundos para que PostgREST recargue.
    - NO reiniciar la base de datos completa — solo el schema cache de PostgREST.
    - El `NOTIFY pgrst` es el mecanismo estándar documentado por Supabase.
  </action>
  <verify>
    Ejecutar desde SQL Editor o desde la app:
    ```sql
    SELECT id, diet_type, meals_count, is_template
    FROM nutrition_plans
    LIMIT 1;
    ```
    La query debe ejecutarse sin error de schema cache.
  </verify>
  <done>PostgREST acepta queries con diet_type, meals_count e is_template sin error de schema cache</done>
</task>
